Batch 2: Layer Groups — Architecture Approach

The flat-list assumption is deeply embedded (100+ index-based accesses). How do we add groups?

A

Flat List + Depth Property (Recommended)

Keep layers as a flat list. Add depth: int and is_group: bool properties. A "group" is a special entry in the list — its children are the subsequent items with higher depth, until depth returns to the group's level.

Pros: Minimal changes to existing code. Index-based access still works. Timeline UI just needs indentation. Compositing walks the list with a stack.

Cons: Reordering layers across groups requires care. Not a "pure" tree — conventions enforced by code, not structure.

This is how Aseprite does it internally.

Impact

  • ~20 lines changed in layer.py
  • ~50 lines in flatten_layers
  • ~30 lines in timeline.py (indent)
  • ~20 lines in project.py (serialize depth)
  • Existing tests mostly unchanged
B

Full Tree Structure

Replace list[Layer] with a tree of Layer | LayerGroup nodes. Each group contains a children list. Provide a flatten_to_display_list() for backward compat.

Pros: Structurally correct. Group boundaries enforced by data structure. Clean compositing via recursion.

Cons: Massive refactor. Every index-based access needs migration. Timeline sync across frames becomes complex. Undo/redo must store layer IDs instead of indices. ~500+ lines of changes.

Impact

  • Rewrite layer.py, animation.py
  • Rewrite timeline.py rendering
  • Update all 100+ index accesses in app.py
  • New serialization format
  • Most existing tests break

My strong recommendation is A. It gives you real nested compositing (groups composite their children before blending with the layer below) while keeping the codebase stable. Aseprite — the gold standard — uses this exact approach. Option B is a multi-day rewrite with high regression risk for no user-visible benefit.
